## 2. 文件存储默认配置行为调整 + DB 唯一约束
- 需求：① 默认配置停用就停用，上传时直接报「默认上传配置已被停用」；② is_default 需唯一 DB 约束。

  - `sql/platform_db.sql` 的 `sys_storage_config` 表：新增函数唯一索引
    `UNIQUE INDEX uk_is_default ((CASE WHEN is_default = 1 THEN 1 END))`

## 3. 修复文件上传报 MultipartException: Stream closed
- 现象：`POST /attachment/upload` 报 `Failed to parse multipart servlet request` → `Caused by: java.io.IOException: Stream closed`，位于 `RequestReplaceFilter` 之后、Spring multipart resolver 之前。
- 根因：`RequestReplaceFilter` 用 `Objects.equals(contentType, ContentType.MULTIPART.getValue())`（即精确匹配 `multipart/form-data`）判断是否跳过包装；但真实上传请求的 Content-Type 是 `multipart/form-data; boundary=----xxx`，匹配失败 → 走到第39行把请求包进 `MyServletRequestWrapper`，其构造函数 `IoUtil.readBytes(super.getInputStream())` 把原始流整段读进内存并关闭，后续 `StandardServletMultipartResolver` 再读流即 `Stream closed`。
- 改法（初版，不完整）：`platform-framework/.../filter/RequestReplaceFilter.java` 第35行判断改为前缀匹配 `contentType.toLowerCase().startsWith(ContentType.MULTIPART.getValue())`，让所有 multipart 请求跳过包装、直接放行。
- 验证：framework 模块 `mvn -o -q -pl platform-framework -am compile` BUILD SUCCESS。

## 4. 修复深化：multipart 跳过判断必须在「包装之前」（初版修复无效）
- 现象：初版修复后线上仍报 `Stream closed`，新栈帧 `RequestReplaceFilter.java:37` 显示仍进了 multipart 分支、但流已被读光。
- 真正根因：原 `doFilterInternal` 第25-27行**无条件先** `new MyServletRequestWrapper(request)`（构造函数即 `IoUtil.readBytes` 读流并关流），multipart 的跳过判断在第35行之后才执行——包装动作早于判断，流早已关闭。初版只改了判断分支、没挪动包装位置，无效。
- 改法（终版）：把 `contentType != null && startsWith("multipart")` 的跳过判断**提到包装之前**，只有非 multipart 才 `new MyServletRequestWrapper`。最终逻辑：multipart → 直接 `doFilter(request, response)` 放行（不包装）；否则包装后放行。
- 验证：framework 模块重新 compile BUILD SUCCESS。
- 关键提醒：framework 是 admin 的依赖模块，必须**重新打包 admin 并重启**才能生效；之前两次线上报错都是跑的旧 framework 字节码（栈帧 :37 即初版代码）。

## 5. 本地上传 FileNotFoundException：父目录未创建
- 现象：Stream closed 修复后上传本地存储报 `FileNotFoundException ... \uploads\2026\07\17\xxx.jpg (系统找不到指定的路径。)`，发生在 `LocalFileStorage.upload` 调 `file.transferTo(dest)`。
- 根因：原代码 `new File(basePath + File.separator + datePath + newFileName)` 用相对路径 `./uploads` 拼接并混用 `\` 与 `/`，且靠 `!dest.getParentFile().exists() && !dest.getParentFile().mkdirs()` 判定建目录；相对路径 + 混合分隔符下 `getParentFile().exists()` 判定不稳，transferTo 内部重新解析路径时父目录实际未建出。
- 改法：`platform-starter/.../file/local/LocalFileStorage.java` 抽出 `resolveDest(fileKey)`：用 `new File(config.getBasePath(), fileKey).getAbsoluteFile()` 绝对化，写入前 `Files.createDirectories(dest.getParentFile().toPath())` 强制建父目录；`upload(MultipartFile)`、`upload(File,String)`（分片合并）、`delete` 三处统一走 `resolveDest`。另抽 `buildFileKey(ext)` 生成 `yyyy/MM/dd/uuid.ext`。
- 验证：starter 模块 `mvn -o -q -pl platform-starter -am compile` BUILD SUCCESS。
- 备注（boss 想法）：想放「仓库根路径下的 .uploads」。当前 base_path=`./uploads` 相对路径，随 JVM 工作目录解析（本机解析到 Tomcat work 目录 ROOT 下）。若要稳定落到仓库根，应把 sys_storage_config 的 base_path 改成绝对路径（或保证 admin 从仓库根启动）。仅代码层面已保证目录会自动创建。

## 6. 分片上传并发报「创建分片目录失败」
- 现象：`POST /attachment/chunk/upload` 同一 identifier 的并发分片（chunk 1/2/3 同一毫秒到达）必现 WARN `创建分片目录失败`，但后续 check 接口返回正常（说明部分分片已落盘）。
- 根因：`ChunkUploadComponent.saveChunk` 第50-53行 `if (!dir.exists() && !dir.mkdirs())` 是 TOCTOU 竞态——同一 identifier 的并发请求同时判断 `!dir.exists()` 为真，先到者 `mkdirs()` 建好目录，后到者再 `mkdirs()` 因目录已存在返回 false，触发 `!exists && !mkdirs` → 抛异常。非并发单分片不触发，故普通小文件上传正常、大文件并发分片才暴露。
- 改法：`platform-component/.../file/ChunkUploadComponent.java` 改用 `Files.createDirectories(dir.toPath())`（幂等且并发安全：目录已存在不报错、父目录缺失自动建好），异常才包成 `BusinessException`。
- 验证：component 模块 `mvn -o -q -pl platform-component -am compile` BUILD SUCCESS。
- 关键提醒：component 也是 admin 依赖模块，必须**重新打包 admin 并重启**才生效。

## 8. 附件下载 / 预览接口（返回 Response 流）
- 需求：附件需要 download + preview 两个接口，后端直接把文件流写进 `HttpServletResponse`（不返回 JSON）。
- 抽象扩展：`FileStorage` 接口新增 `void download(String fileKey, OutputStream out)`（由各实现自行释放底层客户端/流资源，调用方只传输出流）。四个实现均实现：
  - 本地：`Files.copy(resolveDest(fileKey).toPath(), out)`（resolveDest 已在 #5 修过，绝对化+建父目录）。
  - OSS：`ossClient.getObject(bucket,key).getObjectContent()` 拷到 out，`finally` 中 `shutdown()`。
  - COS：`cosClient.getObject(...).getObjectContent()` 拷到 out，`finally` 中 `shutdown()`。
  - MinIO：`client.getObject(GetObjectArgs.builder().bucket().object(key).build())` 拷到 out（MinioClient 无需 shutdown）。
  - 均对 `fileKey` 空 / 文件不存在 抛 `BusinessException(FILE_UPLOAD_FAIL)`（复用现 enum 码，不新增）；云端资源在 finally 关闭，避免泄漏。
- Controller（`SysAttachmentController`）：新增 `GET /attachment/download`（forceDownload=true → `Content-Disposition: attachment`）与 `GET /attachment/preview`（false → `inline`），均无返回值（void，直接写 response）。
  - 统一私有 `doServe(id, response, forceDownload)`：findById → findById(configId) → fileStorageManager.getStorage(config) → 设 `Content-Type`（取自 att.getContentType()，缺省 `application/octet-stream`）、`Content-Disposition`（RFC5987 `filename*=UTF-8''<encoded>`，中文/空格兼容，`encodeFileName` 用 URLEncoder 且空格转 %20）、`Content-Length`（att.getFileSize()）；再 `storage.download(fileKey, response.getOutputStream())` + flush。body 输出用 try-with-resources 自动关 response 输出流。
  - 新增导入：`jakarta.servlet.http.HttpServletResponse`、`java.io.OutputStream/IOException`、`java.net.URLEncoder`。
- 验证：admin 模块 `mvn -o -q -pl platform-admin -am compile` BUILD SUCCESS。重启 admin 后两个接口可用；预览对图片/PDF 等浏览器内联展示，下载强制保存。
- 设计取舍：选择「后端流式代理」（而非重定向到 fileUrl），因对象存储的 fileUrl 可能非公开/需鉴权，代理统一走后端凭证、对前端透明、且跨存储类型一致。代价是对象存储走一次服务端中转带宽，当前场景可接受。

## 9. 修复下载接口 getWriter() already called 崩溃
- 现象：`/attachment/download` 报 `IllegalStateException: getWriter() has already been called for this response`，且 GlobalExceptionHandler 自己也因同一原因二次报错。
- 根因：`LogAspect.around` 对 `download`/`preview` 用 `truncateArgs(joinPoint.getArgs())` 把**全部入参序列化**入库。这两个方法把 `HttpServletResponse` 当入参，Jackson 把它当 bean 反射调用 `getWriter()`（合法 getter）→ 提前占用响应 Writer 通道；随后 `doServe` 调 `getOutputStream()` 即抛异常。
- 修复：`LogAspect` 新增 `isSkippableArg(Object)`，对 `ServletRequest`/`ServletResponse`/`HttpSession`/`Principal`/`InputStream`/`OutputStream`/`MultipartFile`/`byte[]` 跳过（标记 `[skip]`），不再交给 Jackson 序列化；`truncateArgs` 与 `desensitize` 两处均使用。新增对应 import。
- 验证：admin `mvn -o -pl platform-admin -am compile` BUILD SUCCESS。教训：**凡 Controller 方法带 `HttpServletResponse`/`HttpServletRequest` 入参，日志切面序列化必须跳过，否则流式/下载接口必崩**。
