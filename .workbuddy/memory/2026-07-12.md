# 2026-07-12 工作日志

## 用户管理新增 /info 接口（基本信息 + 角色 + 权限 + 菜单树）

- 新增 `UserInfoVO`（com.platform.admin.vo.user）：聚合 user(UserVO) / roles(List<RoleVO>) / permissions(List<String>) / menus(List<MenuTreeVO>)。
- `SysUserController#info`：`GET /user/info?id=`，查询接口。
  - 基本信息复用 `view()` 逻辑（`sysUserService.findById` → UserVO，填充 statusText / pIds / rIds）。
  - 角色：`sysUserService.listRoleById(id)` → 映射 RoleVO（复用现有 RoleVO，填充 statusText）。
  - 权限+菜单：`Set<Long> roleIds` → `sysRoleService.listMenuByIds(roleIds)`；权限=菜单 perms 非空去重集合；菜单=递归 buildMenuTree（沿用 SysMenuController 同款本地 buildTree，rootId=0）。
  - 边界保护：roleIds 为空时 `menuList = Collections.emptyList()`，避免 `listMenuByIds(空集合)` 生成非法 `IN ()` SQL。
- 权限/角色获取权威来源：`platform-framework/.../security/StpInterfaceImpl`（getPermissionList/getRoleList 用 listRoleById + listMenuByIds，写入 Redis user:auth:perm:/role:）。
- 编译验证：`mvn compile -pl platform-admin -am -DskipTests` 通过。

## 部门管理新增 /sort 接口（仅同级排序）

- 新增 `DeptSortVO`（com.platform.admin.vo.dept）：parentId + ids（无 startOrder，与菜单排序一致）。
- `SysDeptController#sort`：`POST /dept/sort`，与 `SysMenuController#sort` 完全同构——
  - 同级校验：`in(id, ids)` 查出 → size 相等 + 所有 parentId==入参 parentId。
  - 事务内按 `i+1` 赋 displayOrder（`deptComponent.doSomethingInTransactional`）。
  - 排序后 `deptComponent.cleanDeptCache()` 清树缓存。
- 复用点：部门树缓存/事务方法均在 DeptComponent（与 MenuComponent 同风格）。perms 建议 `system:dept:sort`。

## 岗位管理新增 /sort 接口（扁平结构，带 startOrder）

- 岗位是扁平分页结构（无 parentId），排序沿用**角色模式**（带 startOrder，分页各页起始值互不干扰），而非菜单/部门的同级模式。
- 新增 `PostSortVO`（com.platform.admin.vo.post）：startOrder + ids，与 RoleSortVO 完全一致。
- 新增 `PostComponent`（com.platform.component.admin.post）：岗位原本无 component，仅提供 `doSomethingInTransactional` 事务包装（岗位无缓存，故无 clean 方法）。
- `SysPostController#sort`：`POST /post/sort`，事务内按 `startOrder + i` 赋 displayOrder，与 `SysRoleController#sort` 同构。
- 关键约定：**项目所有事务都走 component 的 doSomethingInTransactional，无直接 @Transactional 先例**；给无 component 的模块加事务需先建 XxxComponent。perms 建议 `system:post:sort`。
