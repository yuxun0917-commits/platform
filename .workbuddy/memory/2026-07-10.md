## 为 admin 菜单管理功能添加缓存

参考 `sysDept` 模块的菜单树缓存模式（Component 负责读写 JSON 字符串，Controller 负责序列化/反序列化），为菜单管理引入缓存：

- `RedisConstant` 新增 `MENU_TREE = "sysMenu:tree"` 常量。
- `MenuComponent` 新增 `getMenuTree()` / `setMenuTree(json)` / `cleanMenuTreeCache()`（异步删除，沿用 `@Slf4j` 日志风格）；保留原 `cleanMenuCache()`（清用户权限/角色缓存）。
- `MenuController`：
  - `tree()` 改为 cache-aside：先读 `sysMenu:tree` 缓存，未命中则查库→构建树→写回缓存。
  - `add` / `edit` / `editStatus` / `delete` 写操作后调用 `cleanMenuTreeCache()`；`editStatus`/`delete` 在原有 `cleanMenuCache()` 基础上追加树缓存清除。
  - 移除未使用的 `RedisUtil` 注入字段及 import，改用 `JacksonUtil` + `TypeReference` 序列化（与 DeptController 一致）。

约定要点：缓存 value 以 JSON 字符串存储，`set` 无 TTL 时走全局默认（30min）；缓存 key 统一加 `platform:` 前缀。

## 重构：缓存逻辑下沉到 MenuComponent#getMenuTree

用户要求把 cache-aside 完整放进 `MenuComponent#getMenuTree`（直接拿缓存→未命中查库→回填 Redis→返回），Controller 不再负责缓存。

关键架构约束（重要）：
- `platform-component` → `framework` → `service` → `common`；但 `component` 不依赖 `admin`。
- 因此 `MenuComponent` **可以注入 `MenuService`（经查库）**，但**无法引用 `MenuTreeVO`（在 admin）**，也不能构建树。
- 折中方案：`getMenuTree()` 返回 `List<Menu>`（实体在 common），内部完成 读缓存/查库(listTree)/写缓存，缓存内容为扁平菜单列表 JSON；Controller 仍负责 `Menu`→`MenuTreeVO` 映射、填充 statusText/menuTypeText、递归 buildTree（展示层职责）。
- 删除原 `setMenuTree(String)`，Controller 移除 `JacksonUtil`/`TypeReference` 导入与缓存代码；`tree()` 简化为 `return Result.success(menuComponent.getMenuTree() 构建树)`。

## 菜单管理新增排序接口（仅同级）POST /menu/sort

注意：控制器已统一加 `Sys` 前缀（`SysMenuController` / `SysRoleController` 等），实体为 `SysMenu`（字段 `parentId`/`displayOrder`）。

- 参考 `SysRoleController#sort`（`system:role:sort`）实现，新增 `MenuSortVO`（com.platform.admin.vo.menu）：比角色排序多一个 `parentId` 字段，用于同级校验；另含 `startOrder` + `ids`（按序）。
- `SysMenuController#sort`：先 `lambdaQuery().in(id, ids)` 查出这批菜单，校验 `menus.size()==ids.size()`（无效ID）+ 全部 `parentId` 等于入参 parentId（`Assert.isTrue(..., "仅支持同级排序")`）；再在 `menuComponent.doSomethingInTransactional` 事务内按 `startOrder+index` 赋值 displayOrder；最后 `cleanMenuTreeCache()`。
- 排序赋值/事务模式与角色 sort 完全一致；差异仅在“同级校验”这一步。

## 简化菜单排序接口：移除 startOrder

- 用户指出同级排序无需 `startOrder`（起始排序值）：displayOrder 只决定同父级兄弟节点的相对先后，顺序赋 `1,2,3…` 即可，偏移量不影响排序结果，反而让前端多传一个需计算的值。
- 已移除 `MenuSortVO` 的 `startOrder` 字段；`SysMenuController#sort` 改为按 `i + 1` 依次赋值 displayOrder，同级校验（parentId 一致性）逻辑不变。
- 结论：同级排序接口入参精简为 `parentId` + `ids` 两项。
- 编译验证：`mvn compile -pl platform-admin -am -DskipTests` 通过。
