## 1. 操作日志仅记录 POST 请求

## 2. 仪表盘 health 块用「系统平均负载」替换 JVM 内存
- boss 选定 health 返回"系统平均负载"，不再返回 JVM 堆内存使用率（JVM 是应用层指标，与其他三项系统层指标不类）。
- 改动三处：`DashboardHealthBO` / `DashboardHealthVO`（删 `jvmUsage`，增 `load1`/`load5`/`load15`/`coreCount`）、`DashboardComponent#getHealth()`（OSHI `processor.getSystemLoadAverage(3)` 取 [1/5/15] 分钟负载，CPU 逻辑核心数 `getLogicalProcessorCount()`）。
- 关键坑：Windows 上 OSHI 的 `getSystemLoadAverage` 返回 `-1`（无原生 load avg），采集时对 `<0` 置 `null`，避免前端显示 -1；Linux 部署正常出数。
- 前端契约变更：health 返回 `{cpuUsage, memoryUsage, diskUsage, load1, load5, load15, coreCount}`，旧 `jvmUsage` 字段已废弃。负载是否过载需结合 coreCount 判断（load≈coreCount 为满载，>coreCount 为积压）。
- 验证：admin `mvn -o -pl platform-admin -am compile` BUILD SUCCESS。

## 3. health 负载增加「负载使用率」百分比字段（供进度条）
- boss 要求系统平均负载需要"一个百分比进度条的数据"。负载是绝对值，需结合核心数归一化：负载使用率 = load1 / 核心数 * 100（load==核心数 即满载100%，>核心数 即过载）。
- 改动三处：`DashboardHealthBO`/`DashboardHealthVO` 增 `loadUsage`（Double, %）；`DashboardComponent#getHealth()` 在采集 load1/5/15 后计算 `loadUsage = round(load1 / coreCount * 100)`，load 为 null（Windows）时 loadUsage 也置 null。
- 前端契约：`/dashboard/health` 现返回 `{cpuUsage, memoryUsage, diskUsage, load1, load5, load15, coreCount, loadUsage}`。进度条直接用 `loadUsage`；`loadUsage>100` 表示过载，前端可按需 clamp 进度条宽度并用红色提示；`loadUsage==null` 显示"不支持"。文字明细仍可用 `load1/5/15` 展示"x / y / z（1/5/15分钟）"。
- 验证：admin `mvn -o -pl platform-admin -am compile` BUILD SUCCESS。

## 4. health 块：系统平均负载 → 网络利用率（networkUsage）
- boss 决定去掉刚加的系统平均负载（load1/5/15/coreCount/loadUsage），改成网络利用率 `networkUsage`（%）。
- 改动三处：`DashboardHealthBO`/`DashboardHealthVO`（删 load* 全部字段，增 `networkUsage`）、`DashboardComponent#getHealth()`（OSHI 取 `List<NetworkIF>`，与 CPU 共用同一 1s 采样窗口——开头记各合格网卡收发字节、sleep 后算 delta 吞吐 / 带宽）。
- 口径：`networkUsage` = 各物理网卡「收/发方向峰值」中的最大值，再取所有网卡的最大值（max 口径）；`>100` 表示超额。无合格网卡（回环/速率未知）时返回 `null`。
- **OSHI 6.8.2 API 大坑（已踩平）**：`NetworkIF` 在 `oshi.hardware` 包（非 `oshi.software.os`）；方法名是 `getBytesRecv()`/`getBytesSent()`/`getSpeed()`(bits/s)/`updateAttributes()`，**没有** `getStats()`/`isLoopback()`/`isUp()`。回环过滤改用 `getName().toLowerCase().contains("loopback")` + `getSpeed()<=0`。
- 本机 javap 无法解析 oshi jar（报"找不到类"），确认 API 改用 WebFetch OSHI 官方 apidocs（oshi.ooo）最稳。
- 前端契约：`/dashboard/health` 现返回 `{cpuUsage, memoryUsage, diskUsage, networkUsage}`，旧 load* 字段全部废弃。进度条直接用 `networkUsage`，`null` 显示"不支持"，`>100` 标红。
- 验证：admin `mvn -o -pl platform-admin -am compile` BUILD SUCCESS。

## 5. LogTask：操作/登录日志周期 TRUNCATE（MyBatis-Plus SqlRunner）
- boss 纠正：用 MyBatis-Plus 自带 `com.baomidou.mybatisplus.extension.toolkit.SqlRunner`（非 Hutool）。
- `LogTask.cleanOperateLog()` TRUNCATE `sys_log`（每周一4点）、`cleanLoginLog()` TRUNCATE `sys_login_log`（每月1号4点）；两表无外键引用，MySQL TRUNCATE 安全。
- 实现：`SqlRunner.db().update("TRUNCATE TABLE " + table)`；捕获 `Exception`（**不能 catch SQLException**——MyBatis-Plus 的 update 内部已吞掉 SQLException 不向上抛，catch SQLException 会编译报"相应的 try 语句主体中不能抛出异常"）。表名硬编码无注入风险。
- **MyBatis-Plus 3.5.5 的 SqlRunner API 坑（已踩平，重要）**：
  1. 无 `execute(String)` 方法（编译"找不到符号"）。
  2. `update(String)` 返回 `boolean`（不是 int）。
  3. `update` 不声明 throws SQLException。
  4. 正确调用：`boolean ok = SqlRunner.db().update(sql);`（db() 无参走 MyBatis 全局 SqlSessionFactory，Spring Boot 启动时已就绪；job 是 Spring bean 运行时可用）。
- sys_job 种子：`platform_db.sql` 加 id=6/7（cron `0 0 4 ? * MON` / `0 0 4 1 * ?`，concurrent=0，status=1，misfire=1）；表 AUTO_INCREMENT bump 到 8。运行库幂等脚本 `sql/sys_job_log_cleanup.sql`（INSERT IGNORE）。
- 调度解析：`beanName.method()` 格式已支持（RyTask 有先例），无需改 ScheduleComponent。
- 验证：admin `mvn -o -pl platform-admin -am compile` BUILD SUCCESS（4 次编译迭代才调通 API）。

## 6. 核实 SqlRunner 是否需要 yml「enable-sql-runner」
- boss 查网填资料说需配 `mybatis-plus.global-config.enable-sql-runner: true`。
- 直接翻本地 3.5.5 源码核实（项目用 `mybatis-plus-spring-boot3-starter:3.5.5`，自动配置在 `mybatis-plus-spring-boot-autoconfigure:3.5.5`）：
  - `SqlRunner.db()` → 静态单例 `DEFAULT`，`getSqlSessionFactory()` 无 clazz 时返回 `SqlHelper.FACTORY`。
  - `SqlHelper.FACTORY` 由 `MybatisSqlSessionFactoryBean.getObject()` 在 MP 自动配置建 `SqlSessionFactory` 时自动赋值（`SqlHelper.FACTORY = sqlSessionFactory`），**无需任何开关**。
  - `GlobalConfig.enableSqlRunner`（默认 false）在 3.5.5 的 `MybatisPlusAutoConfiguration` 中**完全未被读取**（源码 grep 无引用 + 自动配置 jar 字节码无 "EnableSqlRunner" 字面量；core jar 里的字面量仅 Lombok 生成的 getter/setter 与无关的 `SqlRunnerInjector` 类名）。即该配置在 3.5.5 是 no-op。
- 结论：`LogTask` 现有 `SqlRunner.db().update("TRUNCATE TABLE ...")` **已正确、无需加 yml**；网上说法对应旧版本/旧行为（注册 SqlRunner bean），本项目静态用法不适用。
- 已更新 MEMORY.md 的 SqlRunner 速记，标注 enableSqlRunner 为 3.5.5 空开关，并补充 3 种替代方案（Mapper @Update / JdbcTemplate / SqlSessionFactory）。

## 8. 限流注解 @RateLimit 不生效（根因：自动配置顺序 + key 生成 bug）
- boss 反馈 `@RateLimit` 注解没有生效（已用于 `CaptchaController` 的验证码接口 `limit=5`）。
- **根因 1（注解完全不生效）**：`RateLimiterAutoConfiguration` 在 starter 的 `AutoConfiguration.imports` 中排在 `RedisAutoConfiguration` 之前，且用 `@ConditionalOnBean(RedisTemplate.class)`。Spring Boot 按列表顺序处理自动配置，`@ConditionalOnBean` 检查时 `RedisTemplate` bean 尚未注册 → 条件失败 → `rateLimiterAspect`（`@Aspect`）bean 从未创建 → 注解无任何效果。
- **根因 2（key 错误）**：原 `getCurrentMethod()` 用 `Thread.getStackTrace()[3].getMethodName()` 恒返回 `"around"`，所有被注解接口共用同一计数桶。
- 修复：
  1. `RateLimiterAutoConfiguration` 加 `@AutoConfigureAfter(RedisAutoConfiguration.class)` + 导入该类；imports 文件把 `RedisAutoConfiguration` 移到 `RateLimiterAutoConfiguration` 之前（双保险）。
  2. `RateLimiterAspect#buildKey(ProceedingJoinPoint, RateLimit)` 改用 `joinPoint.getTarget().getClass().getName() + "#" + joinPoint.getSignature().getName()` 作为方法唯一标识，删除 `getCurrentMethod()`。
- 验证：`mvn -o -pl platform-admin -am compile` BUILD SUCCESS（首编因漏 import `RedisAutoConfiguration` 失败一次，补 import 后过）。
- 经验：`@ConditionalOnBean(其他自动配置定义的 bean)` 必须配合 `@AutoConfigureAfter`，否则按 imports 顺序可能永远 false。

## 7. LogTask 日志打印"结果=false"根因与修正
- boss 反馈：两个清理任务都执行成功，但日志打印 `结果=false`。
- 根因：这是**假阴性**。`SqlRunner.db().update("TRUNCATE TABLE ...")` 内部走 `SqlHelper.retBool(rowCount)`（判定 `result >= 1`），而 TRUNCATE 是 DDL，JDBC `executeUpdate` 返回受影响行数 **0**，故 `retBool(0)` 恒为 `false`。只要没抛异常，说明语句已执行、表实际已被清空——`false` 不能代表失败。
- 修正 `LogTask#truncate`：不再使用 update 的返回值，改为截断后 `SqlRunner.db().selectCount("SELECT COUNT(*) FROM " + table)` 校验剩余行数；`==0` 记 `[LogTask] xxx清理完成（已清空）`，`>0` 记 warn，异常记 error。
- 验证：admin `mvn -o -pl platform-admin -am compile` BUILD SUCCESS。
